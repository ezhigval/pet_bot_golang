# Минимальный жизнеспособный продукт (MVP) — ЗЕЛЕНИАЛ

Цель MVP: протестировать ключевую ценность проекта, собрать первую обратную связь пользователей и заложить фундамент для полного SAAS-продукта.

## Основные блоки MVP

### 1. User Service
- Регистрация и авторизация (email, пароль)
- 2FA (через email)
- Привязка устройства к аккаунту
- Управление профилем пользователя (имя, аватар, настройки)
- Ограничение регистрации с одного устройства
- Уведомления о подозрительной активности

### 2. Core API
- Получение базовой информации о пользователе
- Обработка запросов от фронтенда и мобильного приложения
- Аутентификация и проверка JWT

### 3. База данных
- PostgreSQL
- Таблицы: Users, Devices, Registrations, Sessions
- Основные связи:
    - User → Device(s)
    - User → Session(s)
- Индексы для быстрого поиска по email, UUID устройства, IP

### 4. Frontend Web (React + Vite)
- Вход и регистрация
- Профиль пользователя
- Экран здоровья сервиса (`/health`)
- Основная структура навигации

### 5. Mobile App (Kotlin / Swift)
- Регистрация и вход
- Просмотр профиля
- Поддержка push-уведомлений

### 6. Безопасность и конфиденциальность
- Шифрование персональных данных в базе (AES-256)
- JWT с истечением времени
- Логирование попыток входа и подозрительных действий
- Базовая защита от DDoS и brute-force атак

### 7. Логирование и мониторинг
- Zerolog (backend)
- Проверка статуса сервиса через `/health`
- Возможность масштабирования через микросервисы и Kafka

## Основные сценарии использования MVP

1. **Регистрация пользователя**
    - Пользователь вводит email и пароль.
    - Отправляется email с кодом 2FA.
    - Аккаунт создается, устройство привязывается.
    - Если регистрация с уже зарегистрированного устройства, уведомление и сокращение пробного периода.

2. **Вход пользователя**
    - Email + пароль.
    - 2FA.
    - Создается сессия с JWT.

3. **Управление профилем**
    - Изменение имени, аватара.
    - Просмотр привязанных устройств.
    - Настройка уведомлений и безопасности.

4. **Мониторинг и админка**
    - Отслеживание подозрительных регистраций.
    - Получение уведомлений на email разработчиков.
    - Логи безопасности и действий пользователей.

## Ограничения MVP
- Нет интеграции с платежными системами (будет в полной версии)
- Нет аналитики активности пользователей
- Ограниченное количество функциональных блоков (только ядро системы)

## Цель MVP
- Подтвердить правильность архитектуры микросервисов.
- Проверить ключевые функции регистрации, безопасности и управления пользователем.
- Подготовить базу для подключения остальных блоков: финансы, обучение, здоровье, проекты, рефлексия и т.д.
